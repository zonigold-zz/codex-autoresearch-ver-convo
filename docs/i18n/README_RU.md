## Быстрый старт

**1. Установка:**

```bash
git clone https://github.com/leo-lilinxiao/codex-autoresearch.git
cp -r codex-autoresearch your-project/.agents/skills/codex-autoresearch
```

Или используйте skill installer в Codex:
```text
$skill-installer install https://github.com/leo-lilinxiao/codex-autoresearch
```

**2. Откройте Codex в вашем проекте и скажите, что хотите сделать:**

```text
$codex-autoresearch
Убрать все типы any из TypeScript-кода
```

**3. Codex сканирует, подтверждает и итерирует автономно:**

```
Codex: Найдено 47 вхождений `any` в src/**/*.ts.

       Подтверждено:
       - Цель: устранить все типы any в src/**/*.ts
       - Метрика: количество any (текущее: 47), направление: уменьшение
       - Проверка: grep-подсчёт + tsc --noEmit как страховка

       Требуется уточнить:
       - Работать до полного устранения или ограничить N итерациями?

       Ответьте "go" для старта или скажите, что изменить.

Вы:    go, работай всю ночь.

Codex: Начинаю -- базовая линия: 47. Итерирую до прерывания.
```

Улучшения накапливаются, неудачи откатываются, всё протоколируется.

Другие способы установки см. в [INSTALL.md](../INSTALL.md). Полное руководство см. в [GUIDE.md](../GUIDE.md).

## Что он делает

Skill для Codex, запускающий цикл "изменить-проверить-решить" на вашей кодовой базе. Каждая итерация вносит одно атомарное изменение, проверяет его механической метрикой и сохраняет или отбрасывает результат. Прогресс накапливается в git; неудачи автоматически откатываются. Работает с любым языком, любым фреймворком, любой измеримой целью.

Вдохновлён принципами [autoresearch Карпатого](https://github.com/karpathy/autoresearch), обобщёнными за пределы ML.

### Зачем это нужно

Autoresearch Карпатого доказал, что простой цикл -- изменить, проверить, сохранить или отбросить, повторить -- способен за ночь продвинуть ML-обучение от базовой линии до новых высот. codex-autoresearch обобщает этот цикл на всё в программной инженерии, что имеет числовое выражение. Покрытие тестами, ошибки типизации, задержка производительности, предупреждения линтера -- если есть метрика, можно итерировать автономно.

## Архитектура

```
              +---------------------+
              | Зондирование среды  |  <-- Phase 0: определить CPU/GPU/RAM/тулчейны
              +---------+-----------+
                        |
              +---------v-----------+
              | Возобновить сессию? |  <-- проверить артефакты предыдущего запуска
              +---------+-----------+
                        |
              +---------v-----------+
              | Чтение контекста    |  <-- прочитать scope + файл уроков
              +---------+-----------+
                        |
              +---------v-----------+
              | Установить базовую  |  <-- iteration #0
              |      линию          |
              +---------+-----------+
                        |
         +--------------v--------------+
         |                             |
         |  +----------------------+   |
         |  | Выбрать гипотезу     |   |  <-- учесть уроки + перспективы
         |  | (или N параллельно)  |   |      фильтровать по среде
         |  +---------+------------+   |
         |            |                |
         |  +---------v------------+   |
         |  | Внести ОДНО изменение|   |
         |  +---------+------------+   |
         |            |                |
         |  +---------v------------+   |
         |  | git commit           |   |
         |  +---------+------------+   |
         |            |                |
         |  +---------v------------+   |
         |  | Run Verify + Guard   |   |
         |  +---------+------------+   |
         |            |                |
         |        улучшилось?          |
         |       /         \           |
         |     yes          no         |
         |     /              \        |
         |  +-v------+   +----v-----+ |
         |  |  KEEP  |   | REVERT   | |
         |  |+lesson |   +----+-----+ |
         |  +--+-----+        |       |
         |      \            /         |
         |   +--v----------v---+      |
         |   | Записать результат|     |
         |   +--------+--------+      |
         |            |               |
         |   +--------v--------+      |
         |   | Проверка здоровья|      |  <-- диск, git, здоровье проверки
         |   +--------+--------+      |
         |            |               |
         |     3+ отбросов?           |
         |    /             \         |
         |  no              yes       |
         |  |          +----v-----+   |
         |  |          | REFINE / |   |  <-- эскалация протокола pivot
         |  |          | PIVOT    |   |
         |  |          +----+-----+   |
         |  |               |         |
         +--+------+--------+         |
         |         (повторить)        |
         +----------------------------+
```

Цикл работает до прерывания (без ограничений) или ровно N итераций (ограничение через `Iterations: N`).

**Псевдокод:**

```
PHASE 0: Зондирование среды, проверка возобновления сессии
PHASE 1: Чтение контекста + файла уроков

LOOP (бесконечно или N раз):
  1. Обзор текущего состояния + истории git + журнала результатов + уроков
  2. Выбрать одну гипотезу (применить перспективы, фильтровать по среде)
     -- или N гипотез в параллельном режиме
  3. Внести одно атомарное изменение
  4. git commit (до проверки)
  5. Запустить механическую проверку + guard
  6. Улучшилось -> сохранить (извлечь урок). Ухудшилось -> git reset. Упало -> исправить или пропустить.
  7. Записать результат
  8. Health check (диск, git, здоровье проверки)
  9. При 3+ отбросах -> REFINE; 5+ -> PIVOT; 2 PIVOT -> веб-поиск
  10. Повторить. Никогда не останавливаться. Никогда не спрашивать.
```

## Как это работает

Вы говорите одно предложение. Codex делает всё остальное.

Он сканирует ваш репозиторий, предлагает план, согласовывает с вами, затем итерирует автономно:

| Вы говорите | Что происходит |
|------------|---------------|
| "Повысить покрытие тестами" | Сканирует репо, предлагает метрику, итерирует до цели или прерывания |
| "Исправить 12 падающих тестов" | Определяет ошибки, чинит по одной до нуля |
| "Почему API возвращает 503?" | Ищет первопричину с опровергаемыми гипотезами и доказательствами |
| "Этот код безопасен?" | Запускает аудит STRIDE + OWASP, каждый вывод подкреплён кодом |
| "Выпускай" | Проверяет готовность, генерирует чек-лист, контролируемый релиз |
| "Хочу оптимизировать, но не знаю что измерять" | Анализирует репо, предлагает метрики, генерирует готовую конфигурацию |

За кулисами Codex сопоставляет ваше предложение с одним из 7 специализированных режимов
(loop, plan, debug, fix, security, ship, exec). Вам никогда не нужно выбирать режим --
просто опишите свою цель.

## Что Codex определяет автоматически

Codex выводит всё из вашего предложения и вашего репозитория. Вы никогда не пишете конфигурацию.

| Что нужно | Как получает | Пример |
|-----------|-------------|--------|
| Цель | Ваше предложение | "убрать все типы any" |
| Область | Сканирует структуру репозитория | автоматически находит src/**/*.ts |
| Метрика | Предлагает на основе цели + инструментов | количество any (текущее: 47) |
| Направление | Выводит из "улучшить" / "уменьшить" / "устранить" | уменьшение |
| Команда проверки | Сопоставляет с инструментами репозитория | grep-подсчёт + tsc --noEmit |
| Guard (необязательно) | Предлагает при риске регрессии | npm test |

Перед началом Codex всегда показывает вам, что обнаружил, и просит подтверждения.
Минимум один раунд подтверждения, до пяти при необходимости. Затем вы говорите "go" и можете уйти.
Для действительно автономных прогонов запускайте Codex CLI с настройками approval / sandbox, которые не будут прерывать `git commit` или `git revert`. В одноразовом или заведомо доверенном репозитории самый простой вариант - дать Codex более широкие разрешения.

### Двойная проверка

Две команды выполняют разные функции:

- **Verify** = "Целевая метрика улучшилась?" (измеряет прогресс)
- **Guard** = "Ничего другого не сломалось?" (предотвращает регрессии)

```text
Verify: pytest --cov=src --cov-report=term 2>&1 | grep TOTAL | awk '{print $NF}'   # покрытие выросло?
Guard: npx tsc --noEmit                                                              # типы всё ещё проходят?
```

Если verify проходит, но guard падает, изменение дорабатывается (до 2 попыток), затем откатывается. Файлы, защищённые guard, никогда не изменяются.

## Подробности режимов

### loop

Режим по умолчанию. Итеративная оптимизация измеримой цели.

```
Вы:    Убрать все типы any из TypeScript-кода

Codex: Найдено 47 `any` в src/**/*.ts.

       Подтверждено:
       - Цель: устранить все типы any
       - Метрика: количество any (текущее: 47), направление: уменьшение
       - Проверка: grep-подсчёт, страховка: tsc --noEmit

       Требуется уточнить:
       - Работать до нуля или ограничить N итерациями?

       Ответьте "go" для старта.

Вы:    go, работай всю ночь.
```

### plan

Превращает расплывчатую цель в проверенную конфигурацию цикла.

```
Вы:    Хочу ускорить наш API, но не знаю, с чего начать

Codex: Просканировал репозиторий. API-слой в src/api/, используется Express + Prisma.

       Требуется уточнить:
       - Что важнее: p95-задержка, пропускная способность или холодный старт?
       - Тестировать весь API или конкретные эндпоинты?

       Сообщите ваш приоритет.

Вы:    p95-задержка, эндпоинт /search самый медленный.
```

### debug

Поиск багов на основе доказательств. Каждая итерация проверяет одну опровергаемую гипотезу.

```
Вы:    API случайно возвращает 503 под нагрузкой, причина неизвестна

Codex: Вижу пул соединений в src/db/pool.ts, ограничение частоты в src/middleware/
       и 3 асинхронных обработчика без обработки ошибок.

       Подтверждено:
       - Симптом: периодические 503 при конкурентной нагрузке
       - Область: src/api/, src/db/, src/middleware/

       Требуется уточнить:
       - Когда началось? После конкретного изменения или было всегда?
       - Если найду причину -- сразу исправлять?

       Сообщите контекст.

Вы:    Началось после рефакторинга пула соединений на прошлой неделе. Сначала найди причину.
```

### fix

Итеративное исправление ошибок до нуля.

```
Вы:    pytest падает, 12 тестов сломались после рефакторинга

Codex: Вижу 12 падений в tests/.

       Подтверждено:
       - Цель: сделать все тесты pytest проходящими
       - Текущее состояние: 12 падений

       Требуется уточнить:
       - Добавить страховку ruff, чтобы не вносить проблемы с линтером?
       - Исправить все 12 или сначала сосредоточиться на конкретном модуле?

       Ответьте "go" для старта.

Вы:    Исправить все, да, страховка ruff. Go.
```

### security

Аудит в режиме только для чтения: STRIDE + OWASP + красная команда. Каждое обнаружение подкреплено фрагментом кода.

```
Вы:    Проверь API на уязвимости к инъекциям

Codex: Подтверждено:
       - Область: src/api/ и src/middleware/ (14 файлов)
       - Фокус: SQL-инъекции, XSS, валидация ввода

       Требуется уточнить:
       - Только отчёт или также исправить критические находки?
       - Охватить также аутентификацию/управление доступом?

       Сообщите предпочтения.

Вы:    Сначала отчёт, аутентификацию тоже проверь.
```

### ship

Контролируемая проверка релиза. Автоматически определяет, что вы выпускаете (PR, деплой, релиз).

```
Вы:    Выпускай

Codex: Обнаружено: PR в main, 3 коммита.

       Подтверждено:
       - Тип: код PR
       - Цель: ветка main

       Требуется уточнить:
       - Сначала пробный прогон или сразу боевой?
       - Мониторинг после релиза? (5 мин / 15 мин / пропустить)

       Сообщите предпочтения.

Вы:    Сначала пробный прогон.
```

Подробное описание и расширенные настройки каждого режима см. в [GUIDE.md](../GUIDE.md).

## Комбинирование режимов

Режимы можно последовательно комбинировать:

```
plan  -->  loop              # сформировать конфигурацию, затем выполнить
debug -->  fix               # найти баг, затем исправить
security + fix               # аудит и исправление за один проход
```

## Обучение между запусками

Каждый итеративный запуск, кроме `exec`, извлекает структурированные уроки -- что сработало, что не сработало и почему. Уроки сохраняются в `autoresearch-lessons.md` (без коммита, как журнал результатов) и используются в начале будущих запусков для направления генерации гипотез в сторону проверенных стратегий и прочь от известных тупиков. Режим `exec` может читать существующие уроки, но не создаёт и не обновляет их.

- Положительные уроки после каждой сохранённой итерации
- Стратегические уроки после каждого решения PIVOT
- Итоговые уроки при завершении запуска
- Ёмкость: максимум 50 записей, старые записи сжимаются с временным затуханием

Подробности в `references/lessons-protocol.md`.

## Умное восстановление из застоя

Вместо слепых повторных попыток после неудач цикл использует систему постепенной эскалации:

| Триггер | Действие |
|---------|----------|
| 3 отброса подряд | **REFINE** -- корректировка в рамках текущей стратегии |
| 5 отбросов подряд | **PIVOT** -- отказ от стратегии, попытка принципиально иного подхода |
| 2 PIVOT без улучшения | **Веб-поиск** -- поиск внешних решений |
| 3 PIVOT без улучшения | **Мягкая блокировка** -- предупреждение и продолжение с более смелыми изменениями |

Одно успешное сохранение сбрасывает все счётчики. Подробности в `references/pivot-protocol.md`.

## Параллельные эксперименты

Проверяйте несколько гипотез одновременно с помощью субагентов-воркеров в изолированных git worktree:

```
Оркестратор (основной агент)
  +-- Воркер A (worktree-a) -> гипотеза 1
  +-- Воркер B (worktree-b) -> гипотеза 2
  +-- Воркер C (worktree-c) -> гипотеза 3
```

Оркестратор выбирает лучший результат, объединяет его и отбрасывает остальные. Включите параллельные эксперименты во время мастера настройки, ответив "да". Если worktree не поддерживаются, выполнение переключается на последовательный режим.

Подробности в `references/parallel-experiments-protocol.md`.

## Возобновление сессии

Если Codex обнаруживает ранее прерванный управляемый запуск в интерактивном режиме, он может возобновить работу с последнего согласованного состояния вместо перезапуска. Основной источник восстановления -- `autoresearch-state.json`, компактный снимок состояния, атомарно обновляемый каждую итерацию. В режиме `exec` состояние существует только во временном файле в `/tmp/codex-autoresearch-exec/...`, и workflow `exec` должен явно удалить его перед завершением. Прямое возобновление через отсоединенный контроллер выполнения требует уже существующего `autoresearch-launch.json`; если этого подтвержденного манифеста запуска нет, нужно начать новый run через обычный launch flow.

Приоритет восстановления для интерактивных режимов:

1. **JSON + TSV согласованы, и launch manifest присутствует:** немедленное возобновление, мастер пропущен
2. **JSON валиден, TSV несогласован:** мини-мастер (1 раунд подтверждения)
3. **JSON отсутствует или поврежден, TSV есть:** вспомогательный скрипт восстанавливает удерживаемое состояние для подтверждения, а затем продолжает с новым манифестом запуска
4. **Ничего нет:** новый старт (предыдущие постоянные артефакты управления run архивируются)

См. `references/session-resume-protocol.md`.

## Режим CI/CD (exec)

Неинтерактивный режим для конвейеров автоматизации. Вся конфигурация передаётся заранее -- без мастера, всегда ограничен, вывод в JSON.

```yaml
# Пример для GitHub Actions
- name: Оптимизация Autoresearch
  run: |
    codex exec <<'PROMPT'
    $codex-autoresearch
    Mode: exec
    Goal: Reduce type errors
    Scope: src/**/*.ts
    Metric: type error count
    Direction: lower
    Verify: tsc --noEmit 2>&1 | grep -c error
    Iterations: 20
    PROMPT
```

Коды выхода: 0 = улучшение, 1 = без улучшения, 2 = жёсткая блокировка.

Перед использованием `codex exec` в CI заранее настройте аутентификацию Codex CLI. Для программных запусков предпочтительна аутентификация по API key.

Подробности в `references/exec-workflow.md`.

## Журнал результатов

Каждая итерация записывается в двух взаимодополняющих форматах:

- **`research-results.tsv`** -- полный аудиторский след: одна основная строка на итерацию и, при необходимости, параллельные строки worker
- **`autoresearch-state.json`** -- компактный снимок состояния для быстрого возобновления сессии в интерактивных режимах

```
iteration  commit   metric  delta   status    description
0          a1b2c3d  47      0       baseline  initial any count
1          b2c3d4e  41      -6      keep      replace any in auth module with strict types
2          -        49      +8      discard   generic wrapper introduced new anys
3          c3d4e5f  38      -3      keep      type-narrow API response handlers
```

В режиме `exec` снимок состояния существует только в `/tmp/codex-autoresearch-exec/...`, и workflow `exec` должен явно удалить его перед завершением. Обновляйте эти артефакты через встроенные helper scripts в `<skill-root>/scripts/...`, а не через каталог `scripts/` целевого репозитория.

Оба файла не коммитятся в git. При возобновлении сессии состояние JSON перекрестно проверяется с восстановленной сводкой основных итераций TSV, а не с простым количеством строк. Сводки прогресса выводятся каждые 5 итераций. Ограниченные запуски выводят финальную сводку от базовой линии до лучшего результата.

Эти артефакты состояния управляются helper scripts, поставляемыми вместе со skill. Вызывайте их через путь установленного skill, а не через каталог `scripts/` целевого репозитория. Здесь `<skill-root>` означает каталог, где находится загруженный `SKILL.md`; в типичной repo-local установке это `.agents/skills/codex-autoresearch`.

- `python3 <skill-root>/scripts/autoresearch_init_run.py`
- `python3 <skill-root>/scripts/autoresearch_record_iteration.py`
- `python3 <skill-root>/scripts/autoresearch_resume_check.py`
- `python3 <skill-root>/scripts/autoresearch_select_parallel_batch.py`
- `python3 <skill-root>/scripts/autoresearch_exec_state.py`
- `python3 <skill-root>/scripts/autoresearch_launch_gate.py`
- `python3 <skill-root>/scripts/autoresearch_resume_prompt.py`
- `python3 <skill-root>/scripts/autoresearch_runtime_ctl.py`
- `python3 <skill-root>/scripts/autoresearch_commit_gate.py`
- `python3 <skill-root>/scripts/autoresearch_health_check.py`
- `python3 <skill-root>/scripts/autoresearch_decision.py`
- `python3 <skill-root>/scripts/autoresearch_lessons.py`
- `python3 <skill-root>/scripts/autoresearch_supervisor_status.py`

Для человека теперь остается только одна основная точка входа: **`$codex-autoresearch`**.

- В первом интерактивном запуске опишите цель естественным языком, ответьте на вопросы подтверждения, затем напишите `go`
- После `go` Codex записывает `autoresearch-launch.json` и автоматически запускает отсоединенный контроллер выполнения
- Каждый следующий managed runtime cycle запускает неинтерактивную сессию `codex exec` и передает runtime prompt через stdin
- Последующие запросы `status`, `stop` и `resume` по-прежнему идут через тот же `$codex-autoresearch`
- `Mode: exec` остается продвинутым путем для CI и полностью заданной автоматизации

Прямые команды управления по-прежнему доступны для скриптов и отладки выполнения:

- `python3 <skill-root>/scripts/autoresearch_runtime_ctl.py status --repo <repo>`
- `python3 <skill-root>/scripts/autoresearch_runtime_ctl.py stop --repo <repo>`

## Модель безопасности

| Ситуация | Обработка |
|----------|-----------|
| Грязное рабочее дерево | Предпусковая проверка runtime блокирует запуск и перезапуск, пока изменения вне заданной области не будут очищены или изолированы |
| Неудачное изменение | Используется стратегия отката, согласованная до запуска: `git reset --hard HEAD~1` только в изолированной экспериментальной ветке/worktree с одобрением; иначе используется `git revert --no-edit HEAD`; журнал результатов остаётся аудиторским следом |
| Падение guard | До 2 попыток доработки, затем откат |
| Синтаксическая ошибка | Немедленное исправление, не считается итерацией |
| Падение во время выполнения | До 3 попыток исправления, затем пропуск |
| Исчерпание ресурсов | Откат, попытка меньшего варианта |
| Зависание процесса | Завершение по таймауту, откат |
| Застревание (3+ отбросов) | Стратегия REFINE; 5+ отбросов -> PIVOT к новому подходу; при необходимости эскалация к веб-поиску |
| Неопределённость в ходе цикла | Автономно применять лучшие практики; никогда не прерываться для вопроса пользователю |
| Внешние побочные эффекты | Режим `ship` требует явного подтверждения на этапе предпусковой настройки |
| Ограничения среды | Определяются при запуске; неосуществимые гипотезы отфильтровываются автоматически |
| Прерванная сессия | Возобновление с последнего согласованного состояния при следующем вызове |
| Дрейф контекста (длительные запуски) | Проверка отпечатка протокола каждые 10 итераций; повторное чтение с диска при сбое; разделение сессии после 2 компактификаций |

## FAQ

**Как выбрать метрику?** Используйте `Mode: plan` -- он проанализирует кодовую базу и предложит подходящую.

**Поддерживает ли любой язык?** Да. Протокол не зависит от языка. Только команда проверки специфична для предметной области.

**Как остановить?** Прервите Codex или задайте `Iterations: N`. Состояние git всегда консистентно, потому что коммиты происходят до проверки.

**Режим security изменяет код?** Нет. Только анализ в режиме чтения. Скажите Codex "также исправь критические находки" на этапе настройки, чтобы включить исправление.

**Сколько итераций нужно?** Зависит от задачи. 5 для точечных исправлений, 10-20 для исследования, без ограничений для ночных запусков.

**Обучается ли между запусками?** Да. Уроки извлекаются после каждого `keep`, после каждого `pivot` и при завершении управляемого выполнения, если недавнего урока еще нет. Файл уроков сохраняется между сессиями; `exec` только читает уже существующие уроки.

**Может ли возобновить работу после прерывания?** Да, для управляемых запусков, у которых уже есть `autoresearch-launch.json`, `research-results.tsv` и `autoresearch-state.json`. Если подтвержденный launch state отсутствует, запустите новый run через обычный launch flow.

**Может ли искать в интернете?** Да, при застревании после нескольких смен стратегии. Результаты веб-поиска рассматриваются как гипотезы и проверяются механически.

**Как использовать в CI?** Используйте `Mode: exec` или `codex exec`. Вся конфигурация передаётся заранее, вывод в JSON, коды выхода показывают успех/неудачу.

**Может ли тестировать несколько идей одновременно?** Да. Включите параллельные эксперименты на этапе настройки. Используются git worktree для одновременной проверки до 3 гипотез.
